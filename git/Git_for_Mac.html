    Daniel V. ProrokSignature Track· 3 days ago 
    If you have XCode installed on your Mac, the following might be worth a read.

    Not sure if it's relevant, but I'm running MacOS 10.10.1 (Yosemite) and using Safari 8.0.2.

    Issue 1: the auto-download at http://git-scm.com/download/mac didn't work for me. The site says the latest built version (with installer) is 2.0.1 (six months old) and if you want anything newer, you'll need to compile from source. Nevertheless, when I click the "click here to download manually" link, it takes me to http://sourceforge.net/projects/git-osx-installer/files/ which includes a link to a version 2.2.1 installer that was released just a few weeks ago on Dec 21.

    By default, the installer installs git at /usr/local/git/bin/git.

    When I run 
    which git
    I am told it's in /usr/bin/git...great...but...when I ran
    git --version
    I get "git version 1.9.3 (Apple Git-50)"

    Hmmm...that's not the 2.2.1 I just installed.

    /usr/local/git/bin/git --version
    tells me "git version 2.2.1" Cool. But where did this Apple git come from?

    Also, I'd be okay with this, but the "Apple git" doesn't produce any output when I enter
    git config --list
    so I suspect that may be the first of odd things to occur if I can't get "normal git" installed.

    Does MacOS include an "Apple git" by default? Turns out, no, and the answer is right at the top of the README.md file at http://sourceforge.net/projects/git-osx-installer/files/ 

    ## I have XCode installed (and consequently it's bundled git); how do I get my system to use this version instead?

If you have XCode installed, and want to use a later git, running the following should help:

    sudo mv /usr/bin/git /usr/bin/git-xcode
    sudo ln -sf /usr/local/git/bin/git /usr/bin/git

    Bingo! I have XCode installed and that's where Apple git came from. In case you don't know, the two sudo commands above do the following: The first renames the originally installed Apple version of git to "git-xcode". The second symbolically links the /usr/bin/git to the version I just installed at /usr/local/git/bin/git. Now everything seems to be happy.


    Sorry for the long read, but hopefully it helps anyone else who stumbled for a while due to having XCode installed.

    Daniel V. ProrokSignature Track· a day ago 
    There *probably* isn't any problem with XCode using the newer version of git, but if there is, you can always repoint the symbolic link back to the XCode version instead of the one you just installed. You could do this with these commands:

    sudo rm /usr/bin/git
sudo ln -sf /usr/bin/git-xcode /usr/bin/git

    Alternatively, you could forego the symbolic links altogether and fully undo the original renaming and linking procedure with:

    sudo rm /usr/bin/git
sudo mv /usr/bin/git-xcode /usr/bin/git

    Hope that helps!

    Daniel V. ProrokSignature Track· a day ago 
    If you don't have XCode, you shouldn't have to resort to an alias to use git on the command line.

    There's a shell script that ships with the installer that adds /usr/local/git/bin to your path which should make git accessible from the command line automagically. The reason this doesn't work with XCode on the system is that the system finds (and uses) the Apple git in /usr/bin before it gets to searching /usr/local/git/bin for the version you just installed.

    Daniel V. ProrokSignature Track· a day ago 
    Kerimbekov, if you don't have XCode installed, it shouldn't matter where /usr/local/git/bin is in your path. If bash doesn't find it in /usr/bin, it'll keep looking until it eventually finds it in /usr/local/git/bin. If you do have XCode installed, it's probably best to use the "rename the original and create a symbolic link" method above. Depending on path order is sketchy because the path can change under-the-covers and then break things in frustrating-to-figure-out ways.

    Unless MacOS does something I'm not aware of, aliases aren't persistent. As soon as you close your terminal window and open a new one, the alias will be gone. To make an alias persistent so it's there for every terminal you open, you'd need to add it to your ~/.bashrc file. So, deleting an alias is probably as easy as closing the terminal window and opening a new one or, at worst, editing your ~/.bashrc file to remove any alias you entered there.

    Daniel V. ProrokSignature Track· 16 hours ago 
    Anne,

    In short, it sounds like everything is as expected.

    git config --list
    lists all of the git configuration parameters. When you install git for the first time, it has a bunch of default parameters already set. When you enter the two "git config --global" commands, you are adding two *more* parameters to the already long global parameter list. (Git can set some parameters by default to start by making some good guesses to standard settings common to most or all users, but it obviously has no way of knowing your name an e-mail address until you explicitly tell it what these are.)

    After you "tell" git your name and e-mail address, you enter "git config --list" again to verify git now has the two new parameters in the list (user.name and user.email). You can ignore the other several dozen parameters, but you should search through the list to find "user.name" and "user.email" and verify that they are now set properly. (They shouldn't be hard to find; if you just set them, they're probably at the end of the list.)

    Incidentally, after you create an account on Github later in the course and sync your local git directory with your Github one, you'll have a bunch more parameters like "remote.origin.url" and "branch.master.remote" among others should you do "git config --list" again.

    Hope that helps!
